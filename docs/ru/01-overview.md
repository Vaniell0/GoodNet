# 01 — Обзор проекта

## Что такое GoodNet

**GoodNet** — модульный сетевой фреймворк на C++23. Архитектурная цель: дать инженеру raw-контроль над сетевым стеком при нулевой связности между транспортом и бизнес-логикой. Каждый запущенный экземпляр — симметричный **узел (Node)**: он одновременно слушает входящие соединения и инициирует исходящие. Роль в топологии определяется набором загруженных плагинов, а не кодом ядра.

Проект находится в стадии **alpha**: ядро стабильно, ABI плагинов зафиксирован, TCP-транспорт и базовый Logger-хендлер работают и покрыты тестами.

---

## Производительность (измерено)

Тесты между узлами в разных сетях (Docker), Noise_XX + ChaCha20-Poly1305 + Zstd Level 3:

| Потоки | ОЗУ    | Пропускная способность | Условия |
|--------|--------|------------------------|---------|
| 4      | ~30 МБ | ~10 Гбит/с | случайные данные (несжимаемые) |
| 5      | ~300 МБ | 15–17 Гбит/с | случайные данные |
| N      | —      | ~19 Гбит/с | практический предел (CPU bound) |

Предел ~19 Гбит/с — упор в CPU (шифрование + планировщик Asio), не в сети или в коде ядра. На сжимаемых данных Zstd даёт дополнительный выигрыш за счёт уменьшения объёма шифруемых байт.

## Философия

### 1. Симметрия узлов

Нет жёсткого деления «клиент» / «сервер». Загрузили модуль маршрутизации — получили роутер. Загрузили модуль шифрования и мост — получили защищённый прокси. Один бинарник, разный набор `.so`.

### 2. Разделение транспорта и логики

```
Connector.so   →  "как байты попадают в узел"
Handler.so     →  "что с ними происходит"
```

Смена транспорта — это замена одного `.so`. Хендлер не знает ни про сокеты, ни про протокол нижнего уровня. Он получает уже расшифрованный, верифицированный payload через `handle_message()`.

### 3. Безопасность по умолчанию

Каждое не-localhost соединение обязательно проходит:
1. Взаимную аутентификацию Ed25519 (рукопожатие AUTH)
2. X25519 ECDH + BLAKE2b-256 KDF → `session_key`
3. XSalsa20-Poly1305 шифрование всего трафика

Это не опция — это часть ядра. Отключить нельзя, только использовать localhost-режим для локального IPC.

### 4. Цепочка ответственности в диспетчеризации

Пакеты проходят через хендлеры в порядке приоритета (`plugin_info_t::priority`, 255 = первый). Каждый хендлер возвращает `propagation_t` из `on_message_result()`:

```
PROPAGATION_CONTINUE  → передать следующему хендлеру по приоритету
PROPAGATION_CONSUMED  → остановить цепочку + зафиксировать session affinity
PROPAGATION_REJECT    → дропнуть пакет молча
```

### 5. Верифицируемые плагины

Каждый `.so` поставляется с JSON-манифестом, содержащим SHA-256. `PluginManager` проверяет хэш до `dlopen`. Подменить плагин незаметно невозможно.

### 6. Воспроизводимые сборки

Nix Flakes — все разработчики получают побайтово одинаковую сборку. `flake.lock` фиксирует версии всех зависимостей.

## Где GoodNet занимает нишу

| Инструмент | Проблема |
|---|---|
| **Boost.Asio** | Raw I/O без протокола, крипто, роутинга — всё пишешь сам |
| **ZeroMQ** | Навязывает паттерны PUB/SUB/REQ/REP, прячет транспорт, нет крипто |
| **POCO** | Тяжёлый, много слоёв, C++03 стиль |
| **gRPC** | Protobuf, HTTP/2, облачная ориентация, сложная конфигурация |
| **GoodNet** | Асинхронный I/O + крипто + верифицируемые плагины + Nix |

GoodNet подходит для: p2p приложений, IoT-шлюзов, mesh-сетей, защищённого IPC, образовательных проектов о сетевых протоколах.

---

## Что GoodNet не является

- **Не готовым мессенджером** — только протокольный слой
- **Не заменой ZeroMQ** — нет брокера, нет встроенных паттернов сообщений
- **Не cloud-first** — нет встроенного service discovery, TLS-сертификатов, HTTP API

## Терминология

| Термин | Определение |
|---|---|
| **Node** | Запущенный экземпляр GoodNet (`ConnectionManager` + плагины) |
| **Handler** | Плагин бизнес-логики: получает расшифрованные пакеты |
| **Connector** | Транспортный плагин: управляет сокетами |
| **user_key** | Ed25519 keypair пользователя, переносимый между машинами |
| **device_key** | Ed25519 keypair устройства, детерминирован из hardware ID |
| **ephem_key** | X25519 одноразовый keypair для ECDH, уничтожается после деривации |
| **session_key** | XSalsa20-Poly1305 ключ текущей сессии |
| **strand** | `boost::asio::strand` — сериализованный executor без mutex |
| **wire** | Байты в сети: `header_t` ‖ [encrypted] payload |
| **manifest** | `<plugin>.so.json` — метаданные + SHA-256 хэш плагина |
| **conn_id** | `uint64_t` дескриптор соединения, выдаётся ядром при `on_connect()` |
| **propagation_t** | Результат хендлера в цепочке: CONTINUE / CONSUMED / REJECT |
| **plugin_info_t** | Самоописание плагина: имя, версия, приоритет, capability flags |
| **affinity** | Привязка соединения к конкретному хендлеру после первого CONSUMED |
| **peer_id** | `conn_id` пира, вставляется ядром в `endpoint_t` при каждом dispatch |
