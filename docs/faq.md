# FAQ

Частые вопросы о GoodNet, его позиционировании и технических решениях.

## Зачем P2P?

Интернет изначально проектировался как одноранговая сеть. Модель клиент-сервер и HTTP — это надстройка, которая решала конкретную задачу (доставка документов), но со временем стала единственным способом строить приложения.

Результат: каждое приложение зависит от сервера. Нет сервера — нет приложения. Сервер — это single point of failure, точка цензуры, зависимость от провайдера и постоянная статья расходов.

P2P убирает эту зависимость. Узлы общаются напрямую, данные не проходят через посредника, приложение работает пока работает хотя бы один участник.

## Это Web3 / blockchain?

Нет. Blockchain — это один конкретный сценарий использования P2P (распределённый консенсус). GoodNet — платформа для **любых** P2P-приложений: мессенджеры, файлообмен, mesh-сети, IoT, распределённые вычисления, игровые серверы и всё что можно придумать.

GoodNet не привязан к токенам, консенсусу или конкретному протоколу.

## Чем отличается от libp2p?

libp2p — зрелый проект из экосистемы IPFS. Основные отличия:

- **libp2p прячет транспорт** за многослойными абстракциями (multiaddr, muxer, stream). GoodNet даёт прямой доступ к транспортному слою через plugin API. Нужен нестандартный транспорт — пишешь connector, а не патчишь фреймворк.
- **libp2p навязывает свой стек**: multistream, identify, yamux/mplex. GoodNet навязывает только Noise_XX шифрование — всё остальное разработчик выбирает сам.
- **libp2p ориентирован на Go/JS/Rust.** C++ реализация отсутствует или неполноценна. GoodNet написан на C++23 с C ABI — встраивается в любой стек через FFI.

libp2p — хороший выбор если вы работаете в экосистеме IPFS. GoodNet — если нужна платформа без привязки к конкретному протоколу.

## Почему C++?

Платформа/runtime — это фундамент. Он должен:

- **Работать везде** — от сервера до embedded-устройства. C++ компилируется под любую архитектуру.
- **Встраиваться** — GoodNet Core собирается как shared library (`.so`). Его можно загрузить из C, Python, Java, Rust, Go — любого языка с FFI. Go runtime нельзя загрузить как `.so` в чужой процесс.
- **Не добавлять overhead** — zero-cost abstractions. Handler plugin получает `std::span<const uint8_t>` без копирования. GC-пауз нет.
- **Поддерживать plugin-архитектуру** — `dlopen` / `dlsym` работает с C ABI из коробки. Rust ABI нестабилен, Go не поддерживает `dlopen` для плагинов.

C++23 — это не "древний язык". Это concepts, ranges, `std::span`, `std::expected`, structured bindings. Современный C++ ближе к Rust по выразительности, но с доступом к C ABI и 40-летней экосистеме библиотек.

### SDK на чистом C

Плагины не обязаны быть на C++. SDK предоставляет C ABI (`sdk/types.h`, `sdk/handler.h`, `sdk/connector.h`), поверх которого лежат C++ обёртки (`sdk/cpp/handler.hpp`, `sdk/cpp/connector.hpp`). Плагин можно написать на C, Rust (через `extern "C"`), или любом другом языке.

## Почему не Rust?

Rust — отличный язык, но для plugin-based runtime архитектуры C++ подходит лучше:

- **ABI**: C++ через `extern "C"` даёт стабильный C ABI. Rust ABI нестабилен — плагин, собранный одной версией rustc, может не работать с другой.
- **dlopen**: Rust не поддерживает `dlopen` для загрузки Rust-плагинов. Нужно использовать `extern "C"` FFI — фактически писать тот же C ABI.
- **Экосистема**: libsodium, Boost.Asio, spdlog, zstd — проверенные C/C++ библиотеки с многолетней историей. В Rust эквиваленты существуют, но обёртки добавляют слой.
- **Embedded**: `no_std` Rust ограничен. C++ работает на bare metal без ограничений.

GoodNet приветствует плагины на Rust через C ABI — SDK это поддерживает.
